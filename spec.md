# 岡山大学DS部 総合ダッシュボード プロジェクト要件定義・アーキテクチャ設計書

あなたは本プロジェクトのシニアエンジニアとして、以下の設計に基づき、具体的な実装ステップの構築、コンポーネント設計、および新入生向けテンプレートの作成をサポートしてください。

## 1. プロジェクトの目的と背景
* **目的**: 部室のiPad（サイネージ）に常時表示する総合ダッシュボードの開発。
* **教育的背景**: 新入生数チームがそれぞれ単体アプリ（天気、電車、ニュース等）を「バイブコーディング（AIを活用した直感的開発）」で作成し、それを統合して成功体験を積ませる。
* **技術課題の解決**: **pnpm workspaces によるモノレポ**で管理する。各アプリは `apps/` 以下の別ディレクトリに分離し、shadcn/ui コンポーネントは `packages/ui/` で共通化する。Vercel では各アプリを別プロジェクトとして同一リポジトリを参照させ、デプロイは独立して行う。
* **当面のスコープ**: **天気アプリ + ダッシュボード本体最小構成**を最優先で仕上げ、夏休み前（〜2026 年 7 月）に 24 時間放置テスト合格までを到達目標とする。スケジュールアプリは専属チームが並行開発する。残りウィジェット（電車 / ニュース / 部室人数）と各全画面ページのデザイン仕上げは秋以降。詳細は §9。

## 2. 全体アーキテクチャ（統合方式）
ダッシュボードと各アプリは **iframeを用いた「メイン＆サイドバーレイアウト」** で統合する。JSONによるAPI連携は行わない。

> **方針の根拠**: JSON API方式も検討したが、以下の理由でiframe方式を採用する。①各チームがUIに加えてAPIも実装する必要がなく作業量が少ない。②既存アプリのUI資産をそのまま流用できる。③ダッシュボード側でUIを再実装するコストが不要。**フォールバック**: §8-6 の 24 時間放置テストでヒープ増加が合格基準を超えた場合は、JSON API 方式への切り替えを検討する。

* **画面レイアウト（7:3分割）**:
  * 左側70%: スケジュールアプリを大きくiframe表示（スクロール禁止）。
  * 右側30%: 天気、電車などの他アプリの「ミニウィジェット」を縦に並べてiframe表示。
* **透過オーバーレイリンク（Transparent Overlay Link）による遷移**:
  * 右側のミニウィジェットは、カード全体を Next.js の `<Link href="{各アプリのURL}">` で囲む。
  * 内部の `<iframe>` には `pointer-events: none` を当て、直接の操作を禁止する。
  * タップするとブラウザの標準的な画面遷移で、そのアプリの全画面ページ（`/`）へジャンプする。
  * `postMessage` 等の複雑な親子通信は使用しない。
* **各アプリからの戻り導線**:
  * 各アプリの全画面ページ（`/`）右下に配置する「ダッシュボードへ戻る」ボタンは、環境変数 `NEXT_PUBLIC_DASHBOARD_URL` を参照して通常の `<a>` / `<Link>` で遷移する。

## 3. 各アプリチームの要件（テンプレートリポジトリの仕様）
各チームは **Next.js (App Router) + Tailwind CSS** を使用する。Supabaseの利用は状態管理が必要なアプリ（スケジュールなど）のみ任意とし、必須ではない（天気アプリのように外部APIを叩くだけのものは不要）。以下の2画面を必ず実装する。

1. `/widget` (ダッシュボード埋め込み用):
   * 縦横比固定（例: 高さ200px）のウィジェットUI。
   * スクロール禁止（`overflow: hidden`）。
   * `next.config.ts` の `Content-Security-Policy` (frame-ancestors) でダッシュボードからのiframe埋め込みを許可すること。許可ドメインは環境変数 `process.env.NEXT_PUBLIC_DASHBOARD_URL` から動的に生成し、ハードコードしない（ローカル開発と本番でURLが異なるため）。
   * **Server Components のみ**で構成する。`'use client'` ディレクティブは禁止（iPad Safari のメモリフットプリント削減のため）。
   * 自動更新は `<meta http-equiv="refresh" content="60">` を使用。`setInterval` / `setTimeout` による JS タイマーは禁止。
   * 依存ライブラリは最小限にする。`/widget` 側での新規 npm パッケージ追加は原則却下し、必要な場合は上級生レビューを経ること。
2. `/` (全画面メインアプリ):
   * 遷移後に表示されるリッチなメイン画面。新入生がAIを使って自由に機能開発（バイブコーディング）を行う。
   * 画面右下に必ず「ダッシュボードへ戻る」共通ボタン（テンプレートに同梱）を配置すること。遷移先は `NEXT_PUBLIC_DASHBOARD_URL`。

## 4. AI駆動開発の暴走防止ルール（GEMINI.md / .cursorrules）
新入生がAI（Gemini等）を用いて開発する際、デザインの統一感を破壊させないため、テンプレートリポジトリのルートに以下のルールを明記したファイルを配置する。AIエディタはこのルールを絶対厳守すること。

```text
# AIアシスタントへの絶対的な制約事項（Rule）

1. 【コンポーネントの保護】 `components/ui/` 配下のファイルは絶対に編集・削除しないこと。
2. 【UI実装のルール】 標準のHTMLタグ（<button>, <input> 等）に直接Tailwindクラスを当てて独自のUIを作ることを固く禁ずる。必ず `components/ui/` から提供されている既存のコンポーネントをインポートして使用すること。
3. 【新規コンポーネントの配置先】 アプリ独自の新しいコンポーネントを作成する場合は、必ず `components/features/` ディレクトリ内に作成すること。`components/ui/` には絶対に新規ファイルを追加しない。
4. 【カラーコードの禁止】 Tailwindのユーティリティクラスで色を直接ハードコードしないこと（例: `text-red-500` や `bg-[#ff0000]` は禁止）。必ず `text-primary`、`bg-secondary`、`text-muted` など、テーマ変数を使用すること。
5. 【パッケージ追加の許可】 新しいnpmパッケージをインストールする構成変更を提案する場合、実行前に必ずユーザーに許可を求めること。
6. 【ダッシュボード連携】 `/widget` 画面を作成する際は、親要素の高さと幅に100%追従し、スクロールバーを絶対に出さない実装にすること。
7. 【APIキーの保護】 外部APIのキーやシークレットを含む環境変数には、絶対に `NEXT_PUBLIC_` プレフィックスを付けないこと。付けるとクライアント側に露出し流出する。外部API呼び出しは Route Handler または Server Component でサーバー側から行うこと。
8. 【ウィジェットのメモリ最小化】 `/widget` 画面では `'use client'` ディレクティブを使用せず、Server Components のみで構成すること。自動更新は `<meta http-equiv="refresh">` を使用し、`setInterval` / `setTimeout` による JS タイマーは使用しないこと（iPad Safari のメモリリーク対策のため必須）。
9. 【ウィジェットの非インタラクティブ性】 `/widget` 画面はダッシュボード側から `pointer-events: none` で保護されるため、クリック・タップ・ホバーなど一切の操作が検知されない。そのため `/widget` 画面内にはボタン・リンク・タブ・フォームなどのインタラクティブな要素を絶対に配置せず、純粋な「表示専用」UIとして実装すること。インタラクティブな要素は `/`（全画面ページ）にのみ配置すること。
```

## 5. デプロイ環境とローカル開発

### 5-0. モノレポ構成
```
dsc_dashboard/               ← このリポジトリ（pnpm workspaces ルート）
├── apps/
│   ├── dashboard/           ← ダッシュボード本体（port 3000）
│   ├── schedule/            ← スケジュールアプリ（port 3001）
│   ├── occupancy/           ← 部室人数アプリ（port 3005）
│   ├── weather/             ← 天気アプリ（port 3002）
│   ├── transit/             ← 電車アプリ（port 3003）
│   └── news/                ← ニュースアプリ（port 3004）
├── packages/
│   └── ui/                  ← 共通 shadcn/ui コンポーネント
├── pnpm-workspace.yaml
└── package.json
```

ルートの `pnpm-workspace.yaml`:
```yaml
packages:
  - 'apps/*'
  - 'packages/*'
```

全アプリを一括起動:
```bash
pnpm dev  # ルートで実行すると全 apps/* が並行起動
```

### 5-1. デプロイ先
全アプリ・ダッシュボードとも **Vercel** にデプロイする。同一 GitHub リポジトリに対して Vercel プロジェクトを複数作成し、各プロジェクトの **Root Directory** を対応する `apps/<name>` に設定する。`apps/<name>` 以下のファイルが変更されたときのみ該当プロジェクトが再デプロイされる。

### 5-2. URL命名規則
| アプリ | ローカル開発 | 本番URL |
|---|---|---|
| ダッシュボード | `http://localhost:3000` | `https://dsc-dashboard.vercel.app` |
| スケジュール | `http://localhost:3001` | `https://dsc-schedule.vercel.app` |
| 天気 | `http://localhost:3002` | `https://dsc-weather.vercel.app` |
| 電車 | `http://localhost:3003` | `https://dsc-transit.vercel.app` |
| ニュース | `http://localhost:3004` | `https://dsc-news.vercel.app` |
| 部室人数 | `http://localhost:3005` | `https://dsc-occupancy.vercel.app` |

各アプリの `package.json` で dev ポートを固定：
```json
"scripts": { "dev": "next dev -p 3002" }
```

ルート `package.json` の `dev` スクリプトで全アプリを並行起動する（`turbo dev` または `pnpm -r dev`）。

> **注記**: 上記の `*.vercel.app` サブドメインはグローバル一意で早い者勝ちのため、**着手前に 6 つの希望 URL が Vercel で空いているか確認すること**。既に取得済みであれば `dsc-dashboard-okayama` のようなサフィックスを許容する。確実に所有権を持ちたい場合は運用フェーズでカスタムドメインの利用を推奨する（§9-5 参照）。

### 5-3. 環境変数
各アプリの `.env.local` に以下を設定：
```
NEXT_PUBLIC_DASHBOARD_URL=http://localhost:3000
```
本番ではVercelのプロジェクト設定画面で `https://dsc-dashboard.vercel.app` を登録する。未設定時はビルドエラーにする（起動してから壊れるより早期に気づけるため）。

### 5-4. 実行環境
- **Node.js**: 20 LTS
- **TypeScript**: `strict: true` 必須
- `.nvmrc` に `20` を記載してテンプレートに同梱

## 6. セキュリティ要件

### 6-1. iframe sandbox属性
ダッシュボード側がウィジェットを埋め込む `<iframe>` には以下を必ず付ける：
```html
<iframe sandbox="allow-scripts" src="..." />
```
- `allow-same-origin` は付けない
- `allow-top-navigation` は付けない（親ページの遷移を奪われない）

### 6-2. ダッシュボード自身のクリックジャック対策
ダッシュボードの `next.config.ts` で以下のレスポンスヘッダを設定：
```
X-Frame-Options: DENY
Content-Security-Policy: frame-ancestors 'none'
```

### 6-3. APIキー管理
- 外部API呼び出しは Next.js **Route Handler または Server Component** で行う
- APIキーを格納する環境変数は `NEXT_PUBLIC_` を絶対に付けない（付けるとクライアントに露出する）
- GitHubに `.env.local` を誤コミットしないよう `.gitignore` にデフォルトで含める

### 6-4. HTTPS強制
全アプリHTTPS必須。Vercelが自動で付与する。

### 6-5. シークレット漏洩のCIチェック
`NEXT_PUBLIC_` プレフィックス付きの環境変数に API キー系のキーワード（`API_KEY`, `TOKEN`, `SECRET` 等）が含まれていないか、CI で機械的に検知する。`gitleaks` 等のツール導入を推奨。AI にコードを書かせる都合上、ルール文だけでは漏洩を防ぎきれないため、機械チェックを最終防衛線として必須化する。

## 7. ウィジェット仕様

### 7-1. サイズ
- 幅: 親コンテナの100%（ダッシュボード側サイドバー幅 = 画面30%）
- 高さ: **180px固定**
- 個数: 最大4個（天気・電車・ニュース・部室人数）
- ウィジェット間の縦マージン: 12px

> 本仕様は右サイドバー（画面30%）に並べる**ミニウィジェット用**。左 70% にメイン表示するスケジュールアプリは寸法も埋め込み URL もこれとは別建てで、詳細は §9-3 を参照。

### 7-2. 自動更新（ウィジェット内データ）
- 各ウィジェットの `/widget` ページは `<meta http-equiv="refresh" content="60">` により 60 秒ごとに再読み込みする。`setInterval` / `setTimeout` 等の JS タイマーは禁止（iPad Safari のメモリ節約のため）。
- データは Server Components の `revalidate: 60` と組み合わせ、サーバー側で最新化する。

### 7-3. リロード戦略（メモリリーク対策）
iPad Safari でダッシュボードを長時間表示すると JS ヒープが累積する。以下の二段構えで対策する：

- **ソフトリロード（30 分ごと）**: ダッシュボード側の JS で `iframe.src = iframe.src` を各 iframe に対して実行。親 DOM を保持したまま各ウィジェットだけをリセットすることで、チラつきを抑えつつメモリを定期解放する。
- **全体リロード（累積 12h + 復帰契機）**: ダッシュボード起動からの経過時間が 12 時間を超え、かつ `visibilitychange` で `visible` に復帰した瞬間に `window.location.reload()` を実行する。固定時刻（例: 毎朝 06:00）は採用しない（iPad がスリープ中は JS タイマーが発火しないため）。スリープ復帰の「朝イチ」で自動的にリセットされる設計。

```js
// ダッシュボードのクライアントコンポーネントに配置
const startedAt = Date.now();
document.addEventListener('visibilitychange', () => {
  if (document.visibilityState === 'visible') {
    if ((Date.now() - startedAt) / 3600000 > 12) location.reload();
  }
});
```

### 7-4. 焼き付き対策
iPad 液晶の同一ピクセル表示による焼き付きを防ぐため、ダッシュボード全体に 10 分ごとに `transform: translate(Δx, Δy)` を適用し、±1〜2px の範囲で微小にシフトさせる。視覚的にはほぼ気づかれないが、発光ピクセルがずれるため焼き付きリスクを下げられる。

### 7-5. フォールバックと異常検知
ダッシュボード側に `WidgetFrame` コンポーネントを用意し、各ウィジェットをこれで包む：

- **初期ロード失敗**: 10 秒以内に `onLoad` が発火しない、または `onerror` 発火時はフォールバック UI に切り替え
- **stale データ表示**: ウィジェット側で最後に成功したデータをメモリ保持し、取得失敗時はそのまま表示。右上に `stale HH:MM` バッジで最終更新時刻を示す
- **オフライン検知**: ダッシュボード側で `window.addEventListener('online'/'offline')` を監視し、オフライン時は画面上部に細いバナーを表示
- **通信断 5 分でグレースケール**: オフライン状態が 5 分続いた場合、全 iframe を `filter: grayscale(1)` でグレースケール化し、異常を視覚的に伝える

```tsx
<WidgetFrame src={weatherUrl} fallback="天気情報を取得できません" />
```

### 8-4. iPad運用
- ブラウザ: Safari
- モード: **アクセスガイド**（設定 → アクセシビリティ）でフルスクリーン固定
- 向き: 横向き
- 自動スリープ: オフ（設定 → 画面表示と明るさ → 自動ロック → なし）
- ダッシュボードURLをホーム画面に追加し、アイコン起動することでアドレスバー非表示化
- 運用形態: 部員が朝つけて夜スリープさせる前提。完全電源オフではなくスリープ運用のため、リロード戦略は §7-3 に従う（固定時刻トリガーは不可）

### 8-5. ウィジェットの追加方法（widgets.json マニフェスト）
ダッシュボード本体の `public/widgets.json` に設定を 1 行追加するだけで新ウィジェットを組み込める設計にする。新入生が「自分のデプロイ URL を登録すれば部室に映る」達成体験を得られる。

```json
[
  { "id": "weather", "title": "天気", "url": "https://dsc-weather.vercel.app/widget", "order": 1 },
  { "id": "transit", "title": "電車", "url": "https://dsc-transit.vercel.app/widget", "order": 2 },
  { "id": "schedule", "title": "スケジュール", "url": "https://dsc-schedule.vercel.app/widget", "slot": "main", "linkUrl": "https://www.notion.so/..." }
]
```

スキーマ：

| フィールド | 必須 | 説明 |
|---|---|---|
| `id` | ✅ | 一意識別子（英小文字 + ハイフン） |
| `title` | ✅ | 表示用タイトル（フォールバック UI 等で使用） |
| `url` | ✅ | iframe `src` に渡すウィジェット URL（通常 `*/widget`） |
| `slot` | — | `"sidebar"`（既定）または `"main"`。`"main"` は左 70% メイン領域用で原則 1 件のみ |
| `order` | sidebar のみ必須 | サイドバー内の表示順 |
| `linkUrl` | — | カードタップ時の遷移先。**未指定時は `url` のオリジン（= 各アプリの `/`）へ遷移**。スケジュール → Notion のように、全画面ページ以外を遷移先にしたい場合に指定する |

ダッシュボード側は起動時にこれを読み込み、`slot === "main"` のエントリを左 70% に、`slot === "sidebar"`（または未指定）のエントリを `order` 順で右 30% に並べる。新ウィジェット追加時にダッシュボード本体のコード修正は不要。

### 8-6. 展開前の実機放置テスト
本番展開前に、実運用に近い構成（全ウィジェットを本番デプロイし、iPad Safari で表示）で 24 時間の放置テストを実施する：

- Safari のリモートデバッグ（Mac 側 Safari → 開発メニュー）で JS ヒープサイズを初期・12h 後・24h 後で計測
- **合格基準**: 24h 後のヒープ成長が 200MB 未満、クラッシュ・白画面・目立つチラつきなし
- 合格しない場合は §7-2〜7-4 の対策を見直す

## 9. マイルストーンと開発体制

### 9-1. 並行開発体制
本リポジトリは複数チームが並行開発する。当面の体制は以下：

- **天気アプリ + ダッシュボード本体（最小実装）**: 同一担当者がリードする
- **スケジュールアプリ**: 専属チームが担当する。寸法仕様は §7 のサイドバー用ウィジェットと異なり、§2 の「左 70% メイン」要件に従う（詳細は §9-3）

統合の成立条件は、ダッシュボード本体側で `public/widgets.json` に各チームの Vercel URL を 1 行追加すれば即映る状態を**先行整備**しておくこと。スケジュール側がまだ未完成でも、本体は空 URL またはプレースホルダで動く設計にする。

### 9-2. 夏休み前マイルストーン（〜2026 年 7 月）
天気アプリ + ダッシュボード本体最小構成を本番投入水準まで仕上げる。スコープを以下に限定する：

- **天気アプリ `/widget`**: 現在気温 + 今日 / 明日を含む 3 日分の予報（テキスト + 気温）。装飾は最低限
- **天気アプリ `/`**: 全画面ページのデザイン作り込みは秋以降。最低限のレイアウトと「ダッシュボードへ戻る」ボタンのみ
- **ダッシュボード本体**: 7:3 レイアウトの枠、`widgets.json` 読み込み、`WidgetFrame`（§7-5）、リロード戦略（§7-3）、焼き付き対策（§7-4）の実装。スケジュール左メイン領域は空 URL またはプレースホルダでよい
- **検収（縮小構成 24h テスト）**: §8-6 の 24 時間放置テストを **「天気ウィジェット 1 個 + 残り 3 枠は空 iframe または静的 HTML プレースホルダ + スケジュール左メインも空 iframe / プレースホルダ」** という縮小構成で実施し、合格基準（ヒープ成長 200MB 未満・クラッシュなし・白画面なし・目立つチラつきなし）を満たす。**メモリ累積の主な発生源はダッシュボード本体側の JS（リロード戦略・焼き付き対策・`WidgetFrame` 等）であり、空 iframe でも本体側 JS のリーク検証目的は十分に満たせる**。全 5 アプリでの再検証は §9-5 のスケジュール統合・残りウィジェット追加後に実施する

このマイルストーンで実際に検証したいのは「天気の表示」ではなく、**iframe 埋め込み・CSP・meta refresh 60 秒・iPad Safari でのメモリ挙動**である。機能・見た目の作り込みは秋以降の話と最初に線引きする。

### 9-3. スケジュールアプリの仕様
スケジュールアプリは §2 の「左 70% メイン表示」に配置されるため、§7-1 の「幅 100% × 高さ 180px 固定 × 縦並び」の仕様には従わない。

【実装方針】スケジュールアプリは **Notion のデータベースを Notion API 経由で取得し、こちらで shadcn/ui を用いてカレンダー UI を再構成・表示する Next.js アプリ**として実装する。Notion 自体は他ドメインからの iframe 埋め込みを `X-Frame-Options` で拒否するため、Notion ページを直接埋め込む代替案は技術的に取れない。本リポジトリ内の専属チームが担当し、テンプレートを踏襲する前提で各項目を以下に確定する：

| 項目 | 確定内容 |
|---|---|
| 埋め込み URL | `/widget` を使用（他アプリ・テンプレートと統一） |
| 縦横比・スクロール | 親 slot（左 70% × ダッシュボードの利用可能高さ）に幅・高さ 100% 追従。`overflow: hidden` でスクロール禁止。具体ピクセル寸法はダッシュボード本体のレイアウト計算に従う |
| shadcn/ui デザイントークン | §8-2 の共通トークンを使用。独自テンプレートは作らない |
| 戻るボタン規約 | §8-3 の共通規約に従う。ただし本マイルストーンでは `/` 自体が必須ではない（タップ遷移先は後述のとおり Notion 直行のため） |
| `NEXT_PUBLIC_DASHBOARD_URL` | §5-3 共通。`/` を作る場合のみ戻るボタンで参照 |

【データソース側の補足】Notion API のインテグレーショントークンは **サーバー専用シークレット**として扱い、§6-3 に従って `NEXT_PUBLIC_` を付けない。Notion からの `fetch` は Server Component / Route Handler 側で行う（§4 ルール 7・8）。取得頻度・キャッシュ戦略はスケジュールアプリ実装時に詳細化する。

#### インタラクティビティ方針（A 案で確定）

§4 ルール 9 は「`/widget` は `pointer-events: none` で保護されるためインタラクティブ要素を置くな」と定めている。スケジュールアプリの想定ユースケース（カレンダー表示 → Notion で予定詳細）に対しては、**A 案を採用する**：

- **A 案（採用）**: スケジュールも他ウィジェットと同様に**非インタラクティブ**にする。ダッシュボード側で左メイン領域全体を `<Link href="{linkUrl}">` で包み、タップで**直接 Notion へ遷移**させる。スケジュール iframe は `pointer-events: none` の表示専用。遷移はダッシュボード（親フレーム）側のブラウザ標準遷移なので、§6-1 で iframe sandbox に `allow-top-navigation` を付けない方針はそのまま守れる。サイネージは「触れない前提」で全アプリ統一でき、設計の崩れがない
- **B 案（不採用）**: スケジュール iframe のみ `pointer-events` 有効化の例外を作る案。§6-1 の `allow-top-navigation` 不付与方針との矛盾、および `postMessage` 不採用方針（§2）との整合のために追加設計が必要となるため不採用とした

【遷移先の指定方法】Notion URL は §8-5 の `widgets.json` で `linkUrl` フィールドとして指定する。スケジュールエントリの例は §8-5 を参照。`linkUrl` が指定されたエントリは全アプリ共通の挙動として「タップ時の遷移先を `url` のオリジンではなく `linkUrl` に上書き」される。スケジュール固有の特例ロジックは作らない。

【スケジュールチームへの伝達事項】
- スケジュールアプリ側は**クリック・タップを前提にした UI を作らない**（押下しても親 iframe にイベントが届かないため、押せると誤認させると逆効果）
- スケジュールアプリの `/`（全画面ページ）は本マイルストーンでは**必須ではない**。テンプレート構造の整合上残してよいが、ダッシュボードからの遷移先にはならない（タップは直接 Notion へ向かう）
- 「ダッシュボードへ戻る」ボタン（§8-3）は `/` を作る場合のみ実装すればよい

### 9-4. 天気アプリのデータソース固定
天気アプリは **気象庁の天気予報 JSON** を使用する。

- 岡山県エンドポイント: `https://www.jma.go.jp/bosai/forecast/data/forecast/340000.json`
- 公式・無料・API キー不要
- Server Component から `fetch` し、`revalidate: 60` で再取得する。クライアントからの直接 fetch は禁止（§3-1 / §4 ルール 8）

API キー不要のため、§6-3 のシークレット管理は最初のマイルストーンの対象外。教材としてもまず「Server Component から fetch する」という最小要素に集中できる。Open-Meteo 等への切り替えは本マイルストーン後に判断する。

### 9-5. 秋以降のマイルストーン候補

- 残りウィジェット（電車 / ニュース / 部室人数）の追加
- 天気・各アプリの全画面ページのデザイン仕上げ
- 必要に応じてカスタムドメイン化（§5-2 注記参照）
